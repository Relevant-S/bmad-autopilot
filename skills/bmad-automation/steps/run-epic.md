# /bmad-automation run --epic — epic-loop entry sequence

When the practitioner invokes `/bmad-automation run --epic <epic-id>`, execute the sequential epic loop per Story 15.1. The canonical Python composition is `bmad-autopilot/tools/loud-fail-harness/src/loud_fail_harness/epic_lifecycle.py`'s `run_epic_loop` function — that function structurally encodes the four phases below; this prose IS the LLM-runtime protocol that names what each phase is for and what to surface to the practitioner. It mirrors `steps/run.md`'s prose-names-the-Python-composition style one scope up (epic, not story).

The epic loop drives each contained `ready-for-dev` story through the **UNCHANGED** per-story loop (`steps/run.md`'s six-step sequence → Dev → review-seam → QA → merge-ready/escalated). The `--epic` flag is purely additive: invoking `/bmad-automation run <story-id>` (no `--epic`) reaches the per-story entry point bit-identically to Phase 1 (Story 15.1 AC-2; precedent Story 10.4 AC-5). No new specialist is introduced (the 4-specialist set Dev / Review-BMAD / QA / Review-LAD is PRD-locked); no 4th hook is added.

## The four-phase epic loop

### (1) Enumerate the contained stories

Invoke `enumerate_epic_stories(epic_id, sprint_status_path=...)`. It reads the `sprint-status.yaml` `development_status` slice for `<epic-id>` (e.g. `epic-15` → every `15-*` story key; the `epic-15` / `epic-15-retrospective` keys are excluded by the `<N>-` prefix test), keeps only entries whose status is `ready-for-dev`, and returns them in epic-defined (key-ascending, numeric-ordinal) order so `15-2-*` precedes `15-10-*`.

If enumeration yields zero `ready-for-dev` stories, OR the sprint-status file is missing/malformed, OR `<epic-id>` is not of the form `epic-<N>`: `EpicStoryEnumerationError` is raised. Surface its diagnostic verbatim to the terminal stream and HALT. This is an **entry-time precondition diagnostic** (`marker_class = None`), NOT a loud-fail runtime marker — the same posture as `steps/run.md`'s pre-flight `StoryDocNotFound` / `SprintStatusMismatch` family; no PR bundle is produced when the epic entry halts at enumeration.

### (2) Initialize the epic-run-state cache

Resolve the per-epic retry-budget multiplier from `_bmad/automation/config.yaml` via `retry_budget.resolve_per_epic_retry_budget_multiplier` (or `read_per_epic_retry_budget_multiplier_from_config_file`) — an integer ≥ 1, default 2 (Story 15.2 AC-1; the per-story `retry_budget` is resolved the same way one scope down in `steps/run.md`). A malformed value raises `RetryBudgetConfigError` with the field name + remediation hint (loud-fail; surface verbatim).

Invoke `init_epic_run_state(epic_id, run_id, story_ids, multiplier=<resolved>)` to seed an `EpicRunState` at `current_state="epic-in-progress"`, with `per_story_status` seeded from each story's current lifecycle state (`ready-for-dev` for the enumerated stories), the `per_epic_retry_budget` **structure** populated (`effective_budget = multiplier × story_count`, `consumed = 0`), the `per_epic_cost_partition` zeroed, and `active_markers=()`. (`run_epic_loop` threads the resolved multiplier into `init_epic_run_state` for you.)

Persist it via `advance_epic_run_state(epic_run_state_path, epic_state, transient_marker_classes=..., taxonomy_path=...)` (Story 15.1's epic-scope atomic-write helper — the epic sibling of `run-state.py`'s `advance_run_state`). The canonical on-disk path is `_bmad/automation/epic-run-state.yaml` (`epic_run_state.DEFAULT_EPIC_RUN_STATE_PATH`), co-located with the per-story `_bmad/automation/run-state.yaml` per ADR-009's `_bmad/automation/` umbrella.

**The epic-run-state is a CACHE, not a fourth canonical store (NFR-R8 / ADR-005).** It is an aggregate over the per-story story-docs + `sprint-status.yaml` and is reconstructable from them on recovery; `advance_epic_run_state` performs the cache write only — there is NO story-doc callback at epic scope (the canonical story-doc writes happen one scope down, inside each story's per-story loop). Recovery tiebreak: the story-doc wins.

**Resolve the transient-class set ONCE** (the loop threads it through every `advance_epic_run_state` call). See the re-derivation model below.

### (3) Dispatch each story sequentially

For each story in enumerated order, the loop:

1. Invokes the per-story driver `story_loop_runner(story_id=..., index=M, total=N)`. In production this drives the story through the UNCHANGED per-story loop per `steps/run.md` (locate → validate → branch → init run-state → in-progress → dispatch Dev, then the hook-driven Dev → review-seam → QA progression across resume invocations) and returns a `StoryLoopOutcome` carrying the story's terminal per-story status (`merge-ready` / `done` / `escalated`) **AND** the number of per-story retries that story consumed (read from the per-story `RunState`'s `retry_history`; Story 15.2 AC-3 — chosen over the epic loop reading `DEFAULT_RUN_STATE_PATH` because Epic 18 relocates the per-story run-state per-worktree). **NO change to `orchestrator_run_entry.py`'s public composition, NO change to the per-story `RunState` shape.** The per-story `event_log_appender` (from `steps/run.md` step (e)) is reused UNCHANGED within each story — the epic layer does NOT replace or wrap the per-story stream.
2. Folds the story's terminal status into the epic aggregate via `fold_story_terminal` (updates `per_story_status`, recomputes `current_state` via the PURE `derive_epic_state`).
3. Folds the story's `retries_consumed` into the cumulative `per_epic_retry_budget.consumed`, then applies the per-epic budget via the pure `apply_epic_budget(base_state, consumed, effective_budget, has_undispatched=...)` — layered ON TOP of `derive_epic_state` (budget logic never leaks into `derive_epic_state`). The budget is checked **AFTER** the boundary story reaches terminal — the in-flight story is never interrupted (sensor-not-advisor). If the cumulative `consumed >= effective_budget` with undispatched stories remaining, the epic transitions to `epic-paused-on-budget` and `epic-budget-exhausted` is appended to `active_markers`; escalation keeps `current_state` precedence (the marker still emits additively — AC-6); exhausting the budget exactly on the final story is `epic-complete`, not a pause.
4. Persists the advance via `advance_epic_run_state` (the durable `epic-budget-exhausted` marker survives the transient-marker write-back filter).
5. Surfaces the AC-5 per-epic framing line ("story M of N (`<story-id>`) → `<status>`; epic now `<epic-state>`") via the `progress_sink` — written to the SAME terminal stream the per-story events render to (NFR-O1 live streaming; the epic layer ADDS a framing line at each per-story completion boundary, it does not replace the per-story stream).

**Dispatch mode is config-gated (Story 18.1 / FR-P2-4).** Resolve `parallel_stories` (bool, default `false`) + `max_parallel_stories` (int, default `2`) from `_bmad/automation/config.yaml` via `retry_budget.resolve_parallel_stories` / `resolve_max_parallel_stories` (or their `read_*_from_config_file` siblings; malformed values raise `RetryBudgetConfigError` — surface verbatim).

- **`parallel_stories: false` (the default) — sequential.** Stories run strictly sequentially in epic-defined order, exactly as described above; no worktree is created, no story-file lock is acquired, the per-story run-state uses its existing posture. This path is **bit-identical** to the Phase-1/Epic-15 sequential dispatch (Story 18.1 AC-4).
- **`parallel_stories: true` — concurrent, worktree-isolated.** Phase-3 dispatch delegates to the canonical composition `parallel_dispatch.dispatch_stories_parallel` (`tools/loud-fail-harness/src/loud_fail_harness/parallel_dispatch.py`), which fans out at most `max_parallel_stories` per-story loops concurrently (a bounded thread pool). Each concurrent unit is isolated in its own git **worktree** (`worktree_lifecycle.create_worktree`, ADR-009), wrapped in a **story-file lock** (`story_file_lock.story_file_lock`, Story 14.3) around the per-story BMAD-state writes, and driven against a **per-worktree run-state** addressed via `epic_run_state.worktree_run_state_path(story_id)` (Story 14.4 — each worktree owns a byte-isolated run-state, NOT the shared `DEFAULT_RUN_STATE_PATH`). On terminal, `cleanup_worktree(story_id, preserve_on_escalation=True)` removes a merge-ready worktree (branch preserved per NFR-R3) and PRESERVES an escalated one for human inspection. **Concurrency is at the per-STORY level only**: each story still drives the UNCHANGED per-story loop with its specialists (Dev → Review-BMAD → QA → Review-LAD) **serial within that story** — no within-story specialist parallelism, no 5th specialist. The epic-aggregate folds (`fold_story_terminal` / `apply_epic_budget` / `fold_story_cost` / `advance_epic_run_state`) run on the dispatching (main) thread as units complete — single-writer to `epic-run-state.yaml`.

**The admission invariant holds per concurrent unit (Story 16.5 obligation; AC-5).** Before launching each unit, the per-epic budget is evaluated via the SAME `apply_epic_budget` pure function the sequential path uses; once the cumulative `consumed >= effective_budget` with undispatched stories remaining, NO further unit is admitted — the in-flight units finish (never interrupted — sensor-not-advisor), their terminals fold, and the epic pauses `epic-paused-on-budget` with the durable `epic-budget-exhausted` marker. A post-only check would admit one unit per in-flight slot past the ceiling; the per-unit gate admits exactly the stories the budget covers.

**On the `.git/index.lock` framing.** The sequential-mode rationale once justified itself with "concurrent `git add`/`commit` race on `.git/index.lock`". Under worktree isolation that contention does NOT arise: each worktree owns its own index at `.git/worktrees/<id>/index` (ADR-009 line 802 — "`$GIT_DIR` is per-worktree"), so concurrent commits across worktrees never race on the shared `.git/index.lock`. The worktree primitive is precisely the escape from that contention class — which is why `parallel_stories: true` requires worktree isolation rather than bare concurrent dispatch on a shared `.git/`.

### (4) Pause on escalation OR budget exhaustion (sensor-not-advisor)

If folding a story's terminal status transitions the epic to `epic-paused-on-escalation` (the story reached `escalated`) **OR** the cumulative per-epic retries exhaust the budget with undispatched stories remaining (`epic-paused-on-budget`), the loop STOPS: the downstream stories do NOT auto-advance (Story 15.1 AC-4 / Story 15.2 AC-4). The orchestrator records the pause (terminal stream + the persisted epic-run-state, including the `epic-budget-exhausted` marker on a budget pause; the running/final epic-level PR bundle the Stop hook renders at this boundary — see below) and surfaces it, but does NOT itself decide remediation — **that decision is the human's**. The two pauses drive DIFFERENT remediation: `epic-paused-on-escalation` is a *quality* issue (inspect the escalated story); `epic-paused-on-budget` is a *cost* issue (raise `per_epic_retry_budget_multiplier` or split the epic, then re-dispatch). When a boundary story BOTH escalated AND exhausted the budget, `current_state` is `epic-paused-on-escalation` (the proximate human-actionable signal) and the `epic-budget-exhausted` marker still emits additively (the cost overage is never silently lost). Flow policy that *halts* is in scope; flow policy that *resolves* a pause is not.

On a clean run (every contained story terminal: `merge-ready` / `done`), the epic transitions to `epic-complete` — including the case where the budget is exhausted exactly on the final story (no undispatched stories remain to guard, so it is NOT a pause).

## Transient-marker re-derivation model (AC-6 — the ratified blocker decision)

This is the first story to persist a run-state and re-feed it on the next SessionStart. To keep a recovery-recomputed `worktree-stale-lock` marker from going **sticky** (contradicting NFR-R2 "transient signal, not sticky state"), Story 15.1 ratifies the **re-derivation model**:

- The Orchestrator MUST NOT persist transient condition-markers into any run-state it writes back to disk. `active_markers` in a persisted (epic OR per-worktree) run-state carries ONLY durable markers.
- Transient condition-markers are recomputed each cycle from live state by `evaluate_reattach`, which is left **UNCHANGED** (honoring Story 14.6 AC-7). Epic-scope SessionStart recovery reuses the existing SessionStart hook + `evaluate_reattach` — there is no 4th hook and no recovery-path edit.
- The transient/durable axis is sourced **structurally from the marker taxonomy** (`marker-taxonomy.yaml`'s `lifetime` field: `transient` | `durable`, default `durable`), NOT a hardcoded code-level enumeration. `advance_epic_run_state` (and `advance_worktree_run_state` for any per-worktree write-back the epic loop performs) filters out every marker class whose taxonomy `lifetime` is `transient` before the atomic rename. The filter consults the taxonomy at runtime, so a future transient marker is covered with zero filter edits.

The structural primitive is `epic_run_state.filter_transient_markers`, fed by `reconciler.load_marker_lifetimes`. `worktree-stale-lock` is the sole `transient` entry at this story; durable markers are never stripped (preserving the Story 1.4 marker-permanence rule).

## Epic-scope SessionStart recovery (no 4th hook)

Epic-scope reattachment reuses the existing SessionStart hook + `evaluate_reattach` UNCHANGED. The epic loop's write-back (phase 2 + 3 above) is where the transient-marker filter prevents stickiness; recovery re-derives the live `worktree-stale-lock` verdict as it does today (the marker is a derived view of live filesystem-lock state, not durable history). The Stop hook's epic-scope dispatch (Story 15.3) re-renders the epic-level PR bundle on each resume-driven completion boundary — see "Epic-level PR bundle" below.

## Scope boundaries

- **Budget ENFORCEMENT landed in Story 15.2** (this protocol): the resolved `per_epic_retry_budget_multiplier` is threaded into init, per-story retries fold into `per_epic_retry_budget.consumed`, exhaustion reaches `epic-paused-on-budget` and emits the durable `epic-budget-exhausted` marker into `active_markers`. The per-story retry budget (`retry_budget` / `evaluate_retry_decision`) is UNCHANGED — the per-epic budget is a SEPARATE, cumulative, additive ceiling on top of it.
- **Epic-level PR bundle (Story 15.3 — LANDED).** The running/final epic-level PR bundle is rendered by the EXISTING Stop hook's epic-scope dispatch branch (no 4th hook): when `_bmad/automation/epic-run-state.yaml` is present, `hooks/stop.sh` additionally invokes `python3 -m loud_fail_harness.bundle_assembly_epic` (canonical composition: `bundle_assembly_epic.assemble_epic_bundle`), rendering the per-story status table (each row pointing to that story's own canonical artifact — merge-ready bundle / escalation bundle / live event-stream), the per-story + per-epic cost partition, the retry-budget consumption, and the per-epic loud-fail block (reusing the Story 6.1 renderer) to `_bmad-output/epic-pr-bundles/<epic-id>/<run-id>.md`. The bundle is RUNNING (regenerated at each per-story completion boundary) and FINAL at epic close — the SAME artifact at successive cache states, idempotent per NFR-R1. Per-story costs reach the partition through the `StoryLoopOutcome.cost` channel folded by `fold_story_cost` (AC-3; the epic loop never reads a per-story cost path — Epic-18-worktree-safe). A per-story-only `run <story-id>` invocation never writes `epic-run-state.yaml`, so its per-story bundle behaviour is bit-identical (Story 15.3 AC-5).
- **`status --epic <epic-id>` (Story 15.4 — LANDED).** A read-only epic-status query — NO state advance, NO mutation (FR48 + NFR-O4 at epic scope; zero write surface, even less than the no-args listing's `orphan-run-state-detected`). Canonical composition: `epic_status_command.inspect_epic` (console script `bmad-automation-status-epic`). It renders a VIEW over the EXISTING `epic-run-state.yaml` cache — epic lifecycle state, the per-story status list (cache order), per-epic retry-budget consumption, the per-epic `active_markers` inline, plus per-story marker presence projected from Story 8.4's `status_command.inspect_story` per contained story (NFR-R8 — no fourth canonical store) — and surfaces pointers to the epic-run-state file + the epic-level PR bundle. Undispatched stories degrade gracefully (`(not yet dispatched — no per-story run-state)`). Halts on no-epic-run-state / epic-id-mismatch with a named-invariant diagnostic (NOT a marker), exit 1. The no-args `/bmad-automation status` listing now additionally GROUPS its per-story rows under per-epic headers (non-terminal epic state alongside non-terminal per-story state) — purely additive, byte-identical when no epic run exists. See `steps/status.md` → `## Epic-scope inspection protocol`.
- **Parallel-story dispatch (Story 18.1 — LANDED, config-gated).** When `parallel_stories: true`, phase-3 delegates to `parallel_dispatch.dispatch_stories_parallel` (per-story worktree + story-file lock + per-worktree run-state; specialists serial within a story; the per-epic admission invariant evaluated per concurrent unit). When `false` (the default) the sequential body above runs verbatim — the false path is bit-identical to Epic 15 and creates no worktree / acquires no lock (Story 18.1 AC-4). Cross-EPIC parallelism is OUT of scope (epics still sequence; stories fan out WITHIN an epic). Concurrent-env-provisioning discipline (Story 18.3) builds on this dispatch seam.
- **Cross-story state-pollution detection (Story 18.2 — LANDED, active under `parallel_stories: true`).** While the parallel dispatcher fans stories out, it runs continuous pollution detection on the three ADR-005 Phase-2 shared-state surfaces — (1) **shared port pool** (two live worktrees allocate the same port), (2) **shared evidence root** (two resolve the same evidence subpath), (3) **epic/sprint aggregate run-state** (two worktrees claim the same story-id, OR a lost-update whose pre-image disagrees with the on-disk aggregate). Canonical composition: `parallel_pollution.detect_state_pollution` over a main-thread-owned live-claim registry, wired into `parallel_dispatch.dispatch_stories_parallel` at a frozen cadence: **(1) pre-admission** (the incoming story's claim is checked against the live registry before the unit is admitted) **+ (2) the completion-fold boundary** (`cf.wait(FIRST_COMPLETED)`, the epic-altitude analogue of NFR-O8's between-specialist cadence, plus the aggregate lost-update pre-image guard). On any detected conflict the durable, sub-classified `parallel-story-state-pollution` marker (`shared-port-collision` / `shared-evidence-root-collision` / `aggregate-run-state-cross-write`, carrying the two colliding story-ids + the named surface) is appended to the epic aggregate's `active_markers` and the epic pauses on `epic-paused-on-escalation` — **loud-fail, NO auto-resolution** (sensor-not-advisor): the substrate does NOT reassign the port, re-path the evidence dir, or reconcile the aggregate; admission of further units halts, the in-flight units DRAIN without interruption, and the practitioner inspects the named surface and decides remediation. The marker surfaces in the `status --epic` query and the epic PR bundle (its `{story_id}`/`{conflicting_story_id}`/`{shared_surface}` placeholders resolve from `EpicRunState.marker_contexts`). Detection is config-gated with the parallel path — the sequential path (`parallel_stories: false`) never reaches the dispatcher and is wholly unaffected.
- **Concurrent env-provisioning discipline (Story 18.3 — LANDED, the FR7 extension that ACTIVATES 18.2's detector).** Story 18.1 left `dispatch_stories_parallel`'s `claim_provider` injectable but unfilled, so in production `run_epic_loop` called the dispatcher with no provider and 18.2's detector was **inert** (no claims registered, nothing to sense). Story 18.3 supplies the first production provider — `env_provisioning.ParallelEnvClaimProvider`, constructed by `run_epic_loop` ONLY on the parallel branch and passed as `claim_provider=` (with the disjoint allocator's `release` as `claim_release=`). That single act does double duty: (a) it is the FR7-extension **prevention** discipline — per concurrent story it hands a **disjoint** pre-flight ephemeral port (`env_provisioning.DisjointPortAllocator` re-rolls over `allocate_ephemeral_port()` so no two LIVE stories share a port; loud-fail on exhaustion), a **distinct env namespace** (`story_env_namespace`), and a `$GIT_COMMON_DIR`-rooted evidence subpath, pre-seeding the port + namespace into the story's per-worktree run-state (the carrier `steps/env-provisioning.md` step (iii) consumes); and (b) it **activates** 18.2's detection at runtime — the live-claim registry now receives real claims, so the sensor finally has something to sense. Prevention (18.3) and detection (18.2) compose exactly: 18.3 hands each story a disjoint claim; 18.2 fires loudly if that discipline is ever violated. Per-story teardown (`teardown_env`) and per-story orphan-cleanup (`cleanup_orphan_processes`, one `orphan-process-cleanup` marker per orphan port — NFR-S6) remain the unchanged Story 4.3 seams scoped to the per-worktree run-state; the disjoint port is released at the dispatcher's per-story terminal-cleanup point so a later wave may reuse it. Sequential mode (`parallel_stories: false`) constructs none of this and is bit-identical to Phase 1 (Story 18.3 AC-6).

## Loud-fail discipline

- Enumeration failures surface via `EpicStoryEnumerationError` (`marker_class = None` — entry-time precondition diagnostic; terminal stream, no PR bundle).
- Per-story failures surface via the UNCHANGED per-story loop's named exceptions + loud-fail markers (`steps/run.md` / `steps/dispatch.md`); the epic layer does not intercept them.
- Epic-run-state writes propagate `OSError` (temp-write / atomic-rename failure) unchanged from `advance_epic_run_state`; the prior cache is left intact (never a partial-state file).
