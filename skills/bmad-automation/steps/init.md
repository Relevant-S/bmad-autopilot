## Runtime protocol (LLM-runtime checklist)

When the practitioner invokes `/bmad-automation init [--overwrite-confirmed] [--yes] [--no-sample-story]`, the LLM-runtime executes the following ordered steps. Each step composes a substrate module via direct Python invocation (`uv run python -c "..."` from `bmad-autopilot/tools/loud-fail-harness/`) — Story 7.6's dev's-call is to keep the substrate's surface narrow (no new CLI entry point in `pyproject.toml` at this story's checkpoint; CLI wrap-up may land at Story 7.8 or 7.9 if the benchmark deems it needed).

1. **Parse argv.** Extract three booleans:
   - `override_confirmed = "--overwrite-confirmed" in argv`
   - `secondary_confirmed = "--yes" in argv`
   - `no_sample_story = "--no-sample-story" in argv`

1.5. **Detect project type** (Story 9.2). Construct `DetectionRequest(project_root=<resolved>)`. Invoke `loud_fail_harness.project_type_detection.detect_project_type(request)`. Branch on `outcome.reason`:
   - **`unambiguous`** — bind `detected_project_type = outcome.project_type` and proceed to step 2. The write to `_bmad/automation/config.yaml` lands at step 4's three non-halt branch handlers (see below); detection itself does NOT touch the filesystem beyond reading top-level indicators.
   - **`ambiguous`** OR **`no-indicators`** — print `outcome.diagnostic` verbatim to stderr and HALT init with non-zero exit code. NO marker emission per Story 9.2 AC-3 (the closed-set marker taxonomy v1 is preserved; ambiguous-detection halt is structurally analogous to argv-parse-error — a pre-precondition surface that fails before the `run_state` machinery is online). The diagnostic already enumerates the three resolution options (edit `config.yaml` to set `project_type` explicitly | restructure the project root | file an issue if the auto-detection rule is wrong).

2. **Run preconditions** (Story 7.3). Invoke `loud_fail_harness.init_preconditions.run_init_preconditions(probe_registry, project_type=detected_project_type, marker_registry=<canonical>, run_state=<runtime>)`. If `run.halted is True`, surface the diagnostic via `format_init_diagnostic(run)` and HALT (the precondition halt path emits its own `env-setup-failed` marker per Story 7.3 AC-3; do NOT proceed to the guard).

3. **Evaluate the non-destructive guard** (Story 7.6). Construct `GuardRequest(project_root=<resolved>, override_confirmed=<parsed>, secondary_confirmed=<parsed>, no_sample_story=<parsed>)`. Invoke `loud_fail_harness.init_non_destructive_guard.evaluate_non_destructive_guard(request, run_state=<runtime>, marker_registry=<canonical>)`. The function returns `(GuardOutcome, RunState | None)`.

4. **Branch on `outcome.action`:**

   - **`proceed-fresh`** — no existing user-owned files. Call `sample_story_scaffold.scaffold_sample_story(SampleScaffoldRequest(project_root=..., opt_out=no_sample_story))` (Story 7.4) AND `config_qa_runbook_stub.scaffold_config_qa_runbook_stubs(StubScaffoldRequest(project_root=...))` (Story 7.5). Print each scaffolder's `notes` line. Then call `project_type_detection.write_detected_project_type(project_root, detected_project_type)` (Story 9.2) to record the detected value in `_bmad/automation/config.yaml`; print `WriteResult.action` for transparency.

   - **`preserve-merge`** — existing user-owned content present; the guard probed merge-safety successfully. Resolve `config_path = resolve_config_path(project_root)` and `qa_runbook_path = resolve_qa_runbook_path(project_root)`. Call `init_non_destructive_guard.additively_merge_config(project_root)` ONLY IF `config_path in outcome.existing_files`; call `init_non_destructive_guard.additively_merge_qa_runbook(project_root)` ONLY IF `qa_runbook_path in outcome.existing_files`. SKIP sample-story regeneration entirely (the existing `sample-auto-001.md` is preserved byte-for-byte; no action needed). Print each `MergeResult.action` + `new_keys_added` for transparency. Per AC-2, also emit a one-line note: "Sample story preserved at `<resolved-path>` (existing user-owned content; not regenerated). Pass `--overwrite-confirmed --yes` to reset to canonical defaults." Then call `project_type_detection.write_detected_project_type(project_root, detected_project_type)` (Story 9.2) to record the detected value in `_bmad/automation/config.yaml`; print `WriteResult.action` for transparency (existing-value-wins semantics preserve the practitioner's manual override per Story 7.6's contract).

   - **`overwrite-confirmed`** — full explicit override (`--overwrite-confirmed --yes`). The guard has already written the structured audit-log entry at `outcome.audit_log_path`; print that path. Then call Story 7.4's `scaffold_sample_story` (subject to `--no-sample-story`) AND Story 7.5's `scaffold_config_qa_runbook_stubs` to OVERWRITE existing files at the canonical targets (per their first-write contracts at `sample_story_scaffold.py:308-384` and `config_qa_runbook_stub.py:351-420`). NO marker fires on this branch — per the verbatim epic AC at `epics.md` line 3058 ("**no marker fires on intentional override**") and the deliberate-decision audit row in `docs/extension-audit.md`. Then call `project_type_detection.write_detected_project_type(project_root, detected_project_type)` (Story 9.2) to record the detected value in the freshly-reset `_bmad/automation/config.yaml`; print `WriteResult.action` for transparency.

   - **`halt-would-destroy`** — secondary confirmation missing OR existing config is malformed YAML. Print `outcome.diagnostic` (already enumerates the practitioner's three options per AC-4: back up, merge, explicit override). The marker `init-would-destroy-existing-artifact` has already been registered in `next_run_state` by `evaluate_non_destructive_guard`; HALT with non-zero exit-code so calling automation surfaces the failure.

5. **TEA-boundary orientation** (Story 7.8). Construct `OrientationRequest(project_root=<resolved>, repo_root=None)`. Invoke `loud_fail_harness.tea_boundary_orientation.emit_orientation_if_first_run(request)`. The function returns an `OrientationOutcome`. Branch:
   - **`emit`** — print a separator line (e.g., a blank line + a horizontal-rule line `─` × 60), then print the heading `🧭 TEA boundary orientation (one-time)`, then print `outcome.message_text` verbatim (preserving the `> ` blockquote prefixes), then print another separator line. The visual block must be DISTINCT and PROMINENT — NOT buried after preceding scaffold output (per the verbatim AC at `epics.md:3118` "as a distinct, prominent block (not buried in init output)").
   - **`skip-already-emitted`** — proceed silently. Optionally print a one-line note like `(TEA orientation already shown on previous \`init\`; pass \`--overwrite-confirmed --yes\` to reset.)` — this surfaces the skip choice without rendering the full block.
   - Orientation emission does NOT fire on the `halt-would-destroy` branch (which exits early at step 4); orientation emission DOES fire on `proceed-fresh`, `preserve-merge`, AND `overwrite-confirmed` branches (gated only by the `tea_boundary_orientation_emitted` field; on `overwrite-confirmed`, the config is reset to canonical defaults so the field is `false` again, and orientation re-emits).

6. **Successful exit.** Print a one-line summary naming which branch was taken and the count of files touched; exit with code 0.
