This document is the public audit surface that operationalizes the BMAD-extension discipline for the Automator. Its source authorities are FR64 (`_bmad-output/planning-artifacts/prd.md` line 899 ŌĆö convention classification), FR65 (line 900 ŌĆö public audit document maintenance), NFR-I4 (line 961 ŌĆö BMAD-core-absorption migration behavior), ADR-002 (`_bmad-output/planning-artifacts/architecture.md` lines 99-204 ŌĆö the 3├Ś3 portability matrix that informs the `upstream-proposal` vs `automator-internal` boundary at the architectural level), ADR-003 (architecture.md lines 205-334 ŌĆö and specifically the "Unknown-unknown skip-classes routing" elaboration at lines 279-292, which delegates the operational workflow this doc owns), and Pattern 7 (architecture.md lines 999-1005 ŌĆö story-doc section adherence, which names `## QA Behavioral Plan` as an upstream-proposal story-doc section).

This doc IS authoritative for: the per-convention table; the five canonical principle paragraphs (epic-close marker sweep, remediation-shape, spike-with-bounded-timebox, atomic-vs-aggregated, finding-taxonomy bumps ŌĆö the no-introductions principle); the skip-class recognition workflow; the per-convention-add procedure. The doc is **review-enforced**, not CI-enforced (per ADR-003 line 292) ŌĆö it is a contributor-discipline artifact, not a CI gate.

## Classification taxonomy

Per FR64, every Automator-introduced convention must carry exactly one of three classifications. Multi-value, novel-value, and unclassified entries are forbidden ŌĆö the extension-purity anti-metric (`_bmad-output/planning-artifacts/prd.md` line 103 ŌĆö "Ōēż 20% automator-internal without an explicit revisit condition") measures against this single-value column.

- **`automator-internal`** ŌĆö an implementation detail that lives at the Automator's wrapper layer (orchestrator, hook scripts, run-state schema, envelope fields) and does NOT extend the BMAD-core lifecycle, story-doc template, or upstream conventions. Choose this when the convention is private to the Automator's mechanism and would have no meaning outside it. Examples: orchestrator wrapper flags, internal envelope fields that wrap (without reshaping) existing BMAD-core taxonomies, run-state field naming. Migration plan: `N/A ŌĆö internal-only`. NFR-I4 does not apply (the convention is never absorbed upstream because it has no upstream-relevant surface).
- **`upstream-proposal`** ŌĆö an extension that reshapes a BMAD-core surface (lifecycle states, story-doc sections, architectural patterns) and is a candidate for upstream RFC submission to BMAD core. Choose this when the convention WOULD have meaning in BMAD core and where the architectural pathway is "ship in the Automator first; if BMAD core absorbs it, follow NFR-I4's migration." Migration plan: populated per NFR-I4's four-stage flow ŌĆö **acknowledgment** (BMAD core ships an equivalent feature) ŌåÆ **adapter window** (both paths work simultaneously) ŌåÆ **deprecation** (the Automator's path is marked deprecated with a release-notes flag) ŌåÆ **removal** (the Automator's path is removed in a subsequent minor release).
- **`research-needed`** ŌĆö the classification is genuinely undecided pending upstream decision, primitive stability, or community input. Choose this when neither `automator-internal` nor `upstream-proposal` is honest yet and the team has bounded the research via the spike-with-bounded-timebox pattern (see ┬¦ Research-blocker handling). Migration plan: `pending research conclusion`. The classification MUST resolve to one of the other two before MVP close ŌĆö see the per-release re-audit cadence in ┬¦ Skip-class recognition workflow.

The choice constrains future migration behavior per NFR-I4: classifying a convention as `upstream-proposal` is a commitment to ship adapter behavior when BMAD core absorbs the equivalent feature; classifying as `automator-internal` is a commitment that the convention is never absorbed (and so misclassifying an absorbable convention as `automator-internal` traps the project on a forked path). Honest classification at introduction time is therefore a load-bearing engineering decision, not a documentation chore.

## Skip-class recognition workflow

This subsection elaborates ADR-003 (architecture.md lines 205-334) and specifically codifies the operational workflow for the unknown-unknown skip-class routing path delegated to this doc by ADR-003 lines 279-292.

A newly-recognized skip-class ŌĆö i.e., a runtime skip event the FRs/NFRs did not anticipate at MVP-design time ŌĆö flows through the four-step process below (verbatim from ADR-003):

```
newly recognized skip-class ŌåÆ
  add to marker-taxonomy.yaml ŌåÆ
  add to orchestrator-event.yaml ŌåÆ
  add fixture / synthetic story ŌåÆ
  re-run CI checks (3), (4), (5) ŌåÆ
  merge
```

The cadence: **the audit doc is reviewed at the close of every epic** (per ┬¦ Marker taxonomy vs. audit doc ŌĆö known-knowns / unknown-unknowns split, where the epic-close marker sweep procedural step lives) **and at every release** (per FR65's "with revisit conditions where applicable" framing combined with the extension-purity anti-metric's per-release measurement cadence at PRD line 103). The two cadences compose: epic-close sweeps catch convention drift produced inside the just-finished epic; per-release sweeps catch drift accumulated across multiple epics and any `research-needed` entries whose timebox has expired without resolution.

The routing decision is explicit:

- **Known skip-classes** ŌĆö those named by FRs / NFRs already in scope at MVP-design time ŌĆö flow through `schemas/marker-taxonomy.yaml` (Story 1.4) + `schemas/orchestrator-event.yaml` (Story 1.3) + the harness fixture corpus (Story 1.7). - **Unknown-unknown skip-classes** ŌĆö drift the FRs/NFRs didn't anticipate ŌĆö flow through this doc and are added to the per-convention table per the append-only discipline above. The audit step is review-enforced, not CI-enforced; reviewers verify the classification is honest and the rationale is load-bearing.

The CI-vs-review split per ADR-003 line 292:

> "Layers A/B/C and the five substrate components are CI-enforced; skip-class recognition is review-enforced. The harness completeness invariant is over the known skip-class set under CI; the unknown-unknown layer is review-enforced via audit."

This is the architectural commitment that makes the audit doc load-bearing rather than aspirational: code reading text catches known drift; humans reading code catch unknown drift; the harness's structural limit is named honestly rather than papered over.

## Marker class boundaries ŌĆö the remediation-shape principle

> **"Markers are remediation-shaped, not emission-point-shaped. Two events that share an emission point but require different remediation paths get distinct marker classes ŌĆö the reconciler routes by class, and routing by sub-classification fields breaks the flat-routing abstraction. Worked example: `hook-failed` and `scope-assertion-violation` share the SubagentStop hook's non-zero exit path but differ in remediation (fix the hook script vs review Dev's actual diff against declared scope), so they are distinct classes. Same pattern for `review-layer-failed` (Epic 3) which is distinct from `hook-failed` for the same reason."**

**Contributor-discipline checklist item:** *Does this marker have a remediation path distinct from existing classes? If yes ŌåÆ distinct class. If no ŌåÆ sub-classification of the existing class.*

The principle is verifiable against the existing taxonomy: `hook-failed`, `scope-assertion-violation`, and `review-layer-failed` are all enumerated in `schemas/marker-taxonomy.yaml` as distinct classes precisely because their remediation paths differ ŌĆö fix the hook script (`hook-failed`); review Dev's actual diff against the declared scope (`scope-assertion-violation`); investigate the failed review layer's specific failure mode (`review-layer-failed`). The reconciler routes by class; emission-point-based grouping would have collapsed these into a single marker and lost the remediation-routing signal.

## Marker class boundaries ŌĆö the atomic-vs-aggregated principle

> **"Markers represent atomic failure surfaces, not aggregated conditions across multiple atomic failures. If an event can be decomposed into existing markers, decompose it. New marker classes earn their place by representing a remediation surface that doesn't already have one ŌĆö not by aggregating multiple existing classes under a new umbrella name. Worked example: Epic 7's `init` precondition checks could be aggregated under a hypothetical `init-precondition-failed` umbrella, but each underlying failure (TEA missing, Playwright MCP unavailable, BMAD core wrong version, git unexpected state) maps to existing markers (`env-setup-failed`, `playwright-mcp-unavailable`, etc.) ŌĆö so `init`'s diagnostic layer aggregates the existing markers in named-invariant diagnostic output (per NFR-O5) rather than introducing a new aggregator class. The diagnostic layer's job is aggregation; the marker taxonomy's job is atomic surfaces."**

**Contributor-discipline checklist item:** *Can this event be decomposed into existing markers? If yes ŌåÆ use existing markers + diagnostic-layer aggregation. If no ŌåÆ new class with distinct remediation surface.*

**Apply together with the remediation-shape principle.** A proposed new marker class earns its place ONLY when (i) its remediation path is distinct from existing classes (the remediation-shape principle above) AND (ii) it is not decomposable into a combination of existing classes (this principle). Failing test (i), the proposal routes to sub-classification of an existing class. Failing test (ii), the proposal routes to diagnostic-layer aggregation per NFR-O5. Only when both tests pass is a new marker class warranted.

## Finding-taxonomy bumps ŌĆö the no-introductions principle

> **"BMAD core's finding taxonomy is the canonical source for `bucket` Ōłł `{decision_needed, patch, defer, dismiss}` and `severity` Ōłł `{HIGH, MED, LOW}` per FR27. The Automator does NOT fork this taxonomy. The forward-compatibility loud-fail path at `validate_return_envelope` is the architectural backstop that makes silent drift impossible: a finding carrying an out-of-enum bucket or severity value, or an extra classification field, is rejected at the substrate seam ŌĆö the bump must travel through this audit doc and `envelope.schema.yaml` BEFORE it can survive the gate."**

**Contributor-discipline checklist item:** *Is this a new bucket value, severity value, or finding classification field on the Review-BMAD envelope's `findings`? If yes ŌåÆ record the BMAD-extension event in this section's audit trail BEFORE bumping `envelope.schema.yaml`'s enum or `$defs/finding` shape; default classification is `upstream-proposal` unless the convention is provably wrapper-layer-only.*

The principle is verifiable against the existing artifacts: `envelope.schema.yaml` `$defs/finding` already declares `bucket` and `severity` as strict `bucket` enum (line 129) and `severity` enum (line 132) and `additionalProperties: false` on findings (line 110); the Review-BMAD wrapper at `agents/review-bmad-wrapper.md` documents the bucket ├Ś severity passthrough invariant as a named contract section (Story 3.2's "Finding-taxonomy passthrough discipline"); the substrate's `validate_return_envelope` rejects any envelope whose findings violate the schema. The three artifacts agree at MVP scope. Epic 3 introduces ZERO new taxonomy values ŌĆö the no-introductions invariant is established structurally by the wrapper's passthrough section + the schema's enum constraint, and operationally by THIS subsection (what to do IF a future contributor or upstream release proposes a bump). The per-convention table at lines 27-34 is NOT modified at this story; the existing five seed entries remain in their chronological-introduction order; the append-only discipline holds.

The forward-compatibility loud-fail ordering is **record first, bump second**: when upstream `bmad-code-review` legitimately introduces a new bucket or severity value (e.g., a future `improvement` bucket), the operational response is (1) record the BMAD-extension event in the per-convention table per ┬¦ How to add a new convention, (2) bump `envelope.schema.yaml`'s enum to include the new value, (3) re-run the harness CI sweep so the substrate seam now accepts the new value structurally, (4) update `_CANONICAL_BUCKETS` / `_CANONICAL_SEVERITIES` in `tools/loud-fail-harness/tests/test_review_bmad_wrapper.py` to include the new value. The canonical "MVP completion evidence" artifact lives at [`docs/mvp-completion-evidence.md`](mvp-completion-evidence.md). **Post-MVP per-FR/NFR-add cadence.** When a Phase-1.5 / Phase-2 story adds a new FR or NFR enumerated in `prd.md`, the validator at `tools/loud-fail-harness/src/loud_fail_harness/mvp_completion_evidence.py` will fail CI with a `missing-requirement-row` finding. The dev adding the FR/NFR is responsible for:

1. Adding the corresponding row to `docs/mvp-completion-evidence.md`'s coverage matrix (between the `<!-- coverage-rows:begin -->` and `<!-- coverage-rows:end -->` anchors), populating all five required columns.
2. Capturing the journey evidence under `docs/mvp-completion-evidence/journey-<n>/` if a new artifact is required (e.g., a new envelope shape OR a new bundle variant).
3. Re-running `uv run mvp-completion-evidence` from `bmad-autopilot/tools/loud-fail-harness/` until 0 findings.
4. Committing the artifact + per-journey captures together.

The post-MVP cadence is **review-enforced** (mirrors `extension-audit.md`'s overall posture per ADR-003 line 292) NOT auto-enforced beyond the validator gate's structural assertion that EVERY enumerated FR/NFR has a row. The dev adding the FR is responsible for adding the row; the validator only signals missing rows, never silently auto-generates them.

**Discipline rationale.** The artifact's bus-factor-mitigation function (per `epics.md:3420-3423`) requires that every MVP FR/NFR retains observable-behavior traceability across project lifetime. Without the per-FR/NFR-add cadence, post-MVP additions would erode the artifact's completeness over time; the CI gate makes erosion visible immediately rather than at the next MVP-cut-equivalent event.
