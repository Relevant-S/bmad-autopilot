# Reference Projects (BMAD Agent Development Automator)

Per PRD Success Criteria verbatim (`_bmad-output/planning-artifacts/prd.md` line 475 — "2–3 BMAD projects of varying types (web, API, mobile) run the full loop each release. Concrete reference projects named in the repo README at launch."): this artifact indexes the BMAD reference projects the Automator's release-discipline is empirically evaluated against, with a per-project row pointing at the captured run record(s) for the most recent release. Phase 1 (MVP) shipped per-journey evidence captures via Story 8.7 under `docs/mvp-completion-evidence/journey-{1..4}/`; Phase 1.5 introduces the per-release reference-run pattern under `docs/reference-runs/<story-id>-<project-type>/` (the mobile row, Story 9.6, lands first; Phase 2+ will append additional rows per project type per release).

The discipline rule: append a new row when a new reference project is selected for an upcoming Phase 2+ release type; update the `Latest Run Record` cell in-place when a subsequent release captures a fresher run against an existing reference project.

- **Regeneration discipline** — append-only at the row level: a NEW row is added when a NEW reference project is selected for an upcoming release type (e.g., a `desktop` row when desktop-MCP support lands post-Phase-1.5). The `Latest Run Record` cell is updated in-place per release when a subsequent release captures a fresher run against an EXISTING reference project (e.g., when Phase 2's mobile run captures fresher artifacts against a real maintainer-owned mobile project, that row's `Latest Run Record` pointer migrates from `reference-runs/9-6-mobile/` to `reference-runs/<phase-2-story-id>-mobile/`). The `First Run Story` cell is immutable — it always names the story that captured the first run record for that project type.
